一、总体架构：双路径设计

PlayCanvas 的高斯溅射阴影系统根据前向渲染器的排序方式自动选择不同的阴影实现策略。这一决策由 GSplatDirector 在配置变更时完成，通过 GSplatManager 的 renderMode 位掩码驱动。

  ┌─────────────────────────────────────────────────┐
  │              GSplatDirector  （配置决策中心）         │
  │  renderMode = GSPLAT_FORWARD | GSPLAT_SHADOW  │
  └────────────────────┬────────────────────────────┘
                       │
  ┌────────────────────▼────────────────────────────┐
  │              GSplatManager  （生命周期管理）        │
  │  _syncShadowRenderer() 按需创建 / 销毁          │
  └────────┬──────────────────────┬─────────────────┘
           │                      │
  ┌────────▼──────────┐  ┌────────▼──────────────┐
  │  CPU Sort 路径      │  │  GPU Sort 路径          │
  │  GSplatQuadRenderer │  │  GSplatHybridRenderer   │
  │                      │  │                          │
  │  ✓ 自投射阴影         │  │  ✗ 无法自投射              │
  │  castShadow = true   │  │  isVisibleFunc → false     │
  │  引擎标准 Shadow 管线  │  │                          │
  │  WebGL + WebGPU       │  │  └─── 委托给 ───┐           │
  └────────────────────┘  └──────────────────────┼────┘
                                               │
                               ┌───────────────▼───────────────┐
                               │  GSplatShadowRenderer          │
                               │  GPU Compute Cull             │
                               │  + Indirect Draw              │
                               │  WebGPU Only                  │
                               └───────────────────────────────┘

为什么 GPU 排序路径需要专用阴影渲染器？

Hybrid 渲染器的 splat 可见性完全由GPU compute 管线决定（interval cull → projector → radix sort → indirect draw），引擎 CPU 端无法知道哪些 splat 应该投射阴影。因此 Hybrid 通过 isVisibleFunc 明确拒绝参与引擎阴影管线：

// gsplat-hybrid-renderer.js : 887–897
meshInstance.isVisibleFunc = (camera) => {
    if (thisCamera.camera === camera && (renderMode & GSPLAT_FORWARD)) return true;
    if (camera.node?.name === SHADOWCAMERA_NAME) return false; // ← 拒绝阴影
    return false;
};

CPU Sort（Quad）路径

meshInstance 注册为常规 shadow caster，完全走引擎标准阴影管线：culling → SHADER_SHADOW pass → 引擎自动注入 SHADOW_PASS define 编译阴影变体。

适用范围：WebGL + WebGPU

GPU Sort（Hybrid）路径

每个方向光维护一个轻量 ShadowLightEntry（独立材质 + mesh instance + buffer），通过专用 GPU compute cull 决定可见性，indirect draw 写入 shadow map。

适用范围：仅 WebGPU

二、渲染管线：五阶段帧内流程

Phase 0
Manager 初始化
Director 配置决策；Manager._syncShadowRenderer() 按需创建 GSplatShadowRenderer，仅 GPU-sort + GSPLAT_SHADOW 时

Phase 2

Phase 3 详解：三 Pass GPU 裁剪流水线

1
Pass 1 — Interval Compaction（粗裁剪） 共享 Pass，与 forward renderer 复用候选缓冲区
子步骤说明
1a. compute-gsplat-interval-cull每个 interval 一个线程：读取世界空间包围球 → 变换 → 对 6 个光源视锥体平面做分离轴测试
1b. Prefix Sum对 countBuffer[0..numIntervals] 做独占前缀和，计算每个存活 interval 的输出偏移
1c. compute-gsplat-interval-scatter每个 interval 一个 workgroup：用前缀和偏移将存活 splat ID 压缩写入 compactedSplatIds

输出：compactedSplatIds[] 密集候选列表 + candidateCount

2
Pass 2 — compute-gsplat-shadow-cull（精细裁剪） 平坦调度，256 线程/workgroup，ceil(N/256) workgroups

每个线程对一个候选 splat 执行 fineCull(splatId)，包含三个无可投影测试：

测试条件说明
① 透明度 opacity ≤ alphaClip 与 forward alpha cull + shadow PS alphaClip 一致
② 尺寸 maxScale < worldSizeThreshold 利用正交投影线性特性，CPU 预计算世界空间阈值，无需逐 splat 投影矩阵乘法
③ 视锥体 dot(plane.xyz, center) + plane.w ≤ -splatRadius 使用 3σ 高斯半径（保守上界），对 6 个光源视锥体平面逐面测试

存活者通过两级原子压缩写入 per-light 的 outputIndices[] + countBuffer：workgroup 内先局部 atomicAdd → workgroup leader 一次性全局预留连续空间 → 并行写入

3
Pass 3 — compute-gsplat-shadow-indirect-args 单线程 compute shader 
let count = countBuffer[0];
let instanceCount = (count + INSTANCE_SIZE - 1u) / INSTANCE_SIZE;
indirectDrawArgs[drawSlot] = DrawIndexedIndirectArgs(
    indexCount,      // 固定: 6 × INSTANCE_SIZE
    instanceCount,   // 由 GPU 原子计数计算出
    0u, 0, 0u           // firstIndex, baseVertex, firstInstance
);

将 GPU 原子可见计数转换为 indirect draw 参数，全程不离开 GPU。

三、Shadow Map 生成：直接投影高斯点

关键结论：Shadow map 是直接渲染高斯点云生成的，不是 mesh 代理、不是简化几何体。使用与 forward pass 完全相同的 gsplat 四边形着色器（gsplatVS + gsplatPS），区别仅在于使用光源 view-projection 矩阵且仅输出深度。

绘制时的着色器路径

// gsplat-shadow-renderer.js : _createMaterial()
const material = new ShaderMaterial({
    vertexWGSL:   '#include "gsplatVS"',   // ← 与 forward 完全相同的 VS
    fragmentWGSL: '#include "gsplatPS"',   // ← 与 forward 完全相同的 PS
});

// 引擎在编译 shadow 变体时自动注入 SHADOW_PASS define
// SHADOW_PASS 下 FS 跳过颜色计算，仅输出深度

// 使用共享 quad mesh（6 个顶点，两个三角形组成单位四边形）
const mesh = GSplatResourceBase.createMesh(this.device);

顶点着色器执行流程（Shadow Pass）

  1. 从 indirect count buffer 读取可见 splat 数量，循环实例化四边形顶点
  2. 每个实例根据 outputIndices[instanceId] 从 work-buffer 纹理读取 splat 的中心、旋转、缩放、不透明度
  3. 用光源的 view-projection 矩阵将 splat 中心投影到 shadow map 空间
  4. 计算 3D 高斯协方差 → 2D 屏幕空间投影 → 协方差特征分解 → 四边形角点坐标
  5. alpha 裁剪：clipCorner 收缩四边形以排除亚阈值区域

片段着色器执行流程（Shadow Pass）

  1. 对四边形内每个像素计算 2D 高斯分布值（指数衰减，由协方差矩阵决定）
  2. 乘以 splat 不透明度
  3. 与 alphaClip 阈值比较，低于阈值 discard
  4. 仅写入深度（SHADOW_PASS define 下跳过颜色输出）
与 Forward Pass 的唯一区别：
  • Forward: 相机 view-projection + 输出颜色 + alpha blend
  • Shadow: 光源 view-projection + 仅深度输出（引擎自动注入 SHADOW_PASS）

四、关键算法优化

① 无可投影数学（最大性能优化）

阴影裁剪完全避免了 forward projector 最昂贵的操作 — computeSplatCov（3D→2D 高斯协方差投影 + 特征分解 + 屏幕空间裁剪）。

原理：方向光 shadow camera 是正交投影，世界空间尺寸与投影像素尺寸成线性关系。亚像素阈值在 CPU 端预计算为世界空间阈值：

focal = shadowRes / orthoHeight
t2 = minPixelSize² × 0.5 - 0.3
worldSizeThreshold = sqrt(max(0, t2)) / focal   // CPU 预计算

GPU 端仅需一次 max(scale.x, scale.y, scale.z) 比较，无需任何矩阵乘法。

② 两级原子压缩（Workgroup-Aggregated Atomics）

// Workgroup 内先局部竞争（256 线程 → 1 个 workgroup atomic counter）
var<workgroup> wgCount: atomic<u32>;
if (valid) { localSlot = atomicAdd(&wgCount, 1); }
workgroupBarrier();

// Workgroup leader 一次性全局预留
if (localIdx == 0) { wgBase = atomicAdd(&globalCount[0], atomicLoad(&wgCount)); }
workgroupBarrier();

// 并行写入（无需原子操作）
if (valid) { outputIndices[wgBase + localSlot] = splatId; }

将全局原子竞争从 O(N) → O(N/256)，大幅提升大规模 splat 场景吞吐量。

③ 全 GPU 驻留，零 CPU 回读

可见 splat 计数（GPU 原子操作产出）→ indirect draw buffer → GPU 直接消费绘制。CPU 完全不知道每帧画了多少 splat，消除了 GPU→CPU 同步点。

④ 平坦调度（Flat Dispatch）

let threadIdx = gid.y * (numWorkgroups.x * 256u) + gid.x;

GPU 占用率与场景如何分割为 interval 无关——单个大 AABB 和 10K 节点的八叉树同等饱和 GPU。

⑤ 候选缓冲区复用

compactedSplatIds 候选列表由 GSplatHybridRendererScratch 提供，forward renderer 和 shadow renderer 在帧内不同时间点使用（forward 先完成 → shadow 覆盖），避免重复 GPU 内存分配。

⑥ 保守剔除原则

所有剔除测试采用保守上界：尺寸使用 max(scale.x, scale.y, scale.z) 作为投影范围上界；视锥体使用 3σ 高斯半径（而非 1σ 中心点）。永远不会错误剔除 forward 管线会保留的 splat。

为什么每个 light 需要独立的 Compute 实例？
多个 light 共享同一个 Shader 对象（代码相同），但若共用 Compute 实例（同一 uniform buffer），后续 light 的 dispatch 会覆盖前一个的 frustum planes / draw slot。Per-light Compute 实例隔离了 uniform buffer + bind group。

七、当前限制与待扩展方向

⚠️ 当前限制

  • 仅方向光 — 不支持点光源（需 6-face cubemap shadow map）和聚光灯
  • 仅非级联方向光（numCascades === 1）— 级联阴影不支持
  • 仅 WebGPU — 依赖 compute shader；WebGL 走 CPU sort 自投射路径
  • Hybrid 依赖 interval 结构 — 世界空间包围盒需 CPU 端生成

🔮 潜在扩展方向

  • 支持级联方向光阴影（per-cascade cull）
  • 支持点光源（per-face cull → 6 次 Pass 1+2 → layered indirect draw）
  • Cull shader 合并 fineCull 的 3 个测试到一个早期退出分支
  • Workgroup 共享内存缓存 splat 数据以减少纹理采样延迟
